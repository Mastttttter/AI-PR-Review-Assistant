# AI PR Review 助手 MVP 版本产品需求文档

## 1. 文档信息

| 项目   | 内容                                       |
| ---- | ---------------------------------------- |
| 产品名称 | AI PR Review 助手                          |
| 版本   | MVP v1.0                                 |
| 文档类型 | 产品需求文档                                   |
| 目标用户 | 开发者、代码 Reviewer、项目负责人                    |
| 产品定位 | 面向开发团队的 AI 辅助代码审查工具                      |
| 核心目标 | 帮助团队在 PR 合并前更快理解代码变更、发现风险问题、提升 Review 效率 |

---

## 2. 产品背景

在软件开发协作中，Pull Request / Merge Request 是代码合并前的重要质量控制环节。开发者提交代码后，通常需要由其他成员进行代码审查。

但在实际团队协作中，PR Review 经常存在以下问题：

1. **Reviewer 阅读成本高**
   PR 涉及多个文件和模块时，Reviewer 需要花较长时间理解改动内容。

2. **低级问题重复出现**
   例如空值判断缺失、异常处理不足、测试缺失、命名不规范等。

3. **Review 质量不稳定**
   不同 Reviewer 的经验不同，审查标准不统一。

4. **新人提交代码质量波动较大**
   新成员可能不熟悉项目规范，导致 PR 中出现较多基础问题。

5. **PR 描述不清晰**
   开发者提交 PR 时，可能没有完整说明改动目的、影响范围和风险点。

因此，需要一个 AI PR Review 助手，在人工 Review 之前自动完成基础审查，帮助开发者和 Reviewer 提升效率。

---

## 3. 产品目标

MVP 阶段的目标不是替代人工 Review，而是完成一次基础的 AI 辅助审查。

### 3.1 核心目标

1. 自动分析 PR 代码变更内容。
2. 自动生成 PR 摘要。
3. 自动识别潜在风险问题。
4. 自动给出修改建议。
5. 输出结构化 Review 报告。
6. 支持团队配置少量 Review 规则。

### 3.2 MVP 成功标准

MVP 版本完成后，应能证明：

> 用户提交一段 PR 变更内容后，系统可以自动生成一份有参考价值的代码审查报告，帮助 Reviewer 更快理解 PR 并发现部分潜在问题。

---

## 4. 用户角色

### 4.1 开发者

开发者是 PR 的提交者。

主要需求：

* 在提交 PR 后快速获得自检结果。
* 知道自己的代码是否存在明显问题。
* 获得修改建议。
* 减少被 Reviewer 反复打回的情况。

### 4.2 Reviewer

Reviewer 是负责审查 PR 的团队成员。

主要需求：

* 快速理解 PR 做了什么。
* 快速知道重点风险在哪里。
* 减少基础问题的重复检查。
* 将精力集中在业务逻辑、架构设计和关键风险上。

### 4.3 项目负责人

项目负责人关注团队整体代码质量和协作效率。

主要需求：

* 了解项目近期 PR 质量情况。
* 统一团队 Review 标准。
* 发现团队常见代码问题。
* 降低新人代码接入成本。

---

## 5. MVP 产品范围

### 5.1 MVP 包含功能

MVP 版本包含以下功能：

1. 创建 PR Review 任务
2. 输入 PR 变更内容
3. 生成 PR 摘要
4. 识别代码风险
5. 生成修改建议
6. 输出结构化 Review 报告
7. 配置基础 Review 规则
8. 查看历史 Review 记录
9. 对 AI 建议进行反馈

---

### 5.2 MVP 不包含功能

MVP 阶段暂不包含以下功能：

1. 不支持自动合并 PR。
2. 不支持自动修改并提交代码。
3. 不支持复杂权限系统。
4. 不支持企业级多组织管理。
5. 不支持 IDE 插件。
6. 不支持完整代码仓库知识库。
7. 不支持多人实时协作评论。
8. 不支持复杂数据看板。
9. 不支持绩效考核类功能。
10. 不保证发现所有代码缺陷。

---

## 6. 核心业务流程

### 6.1 基础流程

```text
用户创建 Review 任务
        ↓
输入 PR 标题、描述、代码变更内容
        ↓
系统生成 PR 摘要
        ↓
系统识别风险问题
        ↓
系统生成修改建议
        ↓
用户查看 Review 报告
        ↓
用户根据建议修改代码或提交人工 Review
```

---

### 6.2 开发者使用流程

1. 开发者进入系统。
2. 创建一个新的 PR Review 任务。
3. 填写 PR 标题。
4. 填写 PR 描述。
5. 输入本次代码变更内容。
6. 点击“开始 Review”。
7. 系统生成审查报告。
8. 开发者查看问题列表。
9. 开发者根据建议修改代码。
10. 开发者可标记建议是否有用。

---

### 6.3 Reviewer 使用流程

1. Reviewer 打开某个 PR Review 报告。
2. 查看 AI 生成的 PR 摘要。
3. 查看风险等级。
4. 查看重点问题列表。
5. 查看具体代码建议。
6. 根据 AI 报告决定人工 Review 重点。
7. 对无效建议进行反馈。

---

## 7. 功能需求

## 7.1 创建 PR Review 任务

### 功能说明

用户可以创建一个新的 PR Review 任务，用于提交代码变更内容并触发 AI 审查。

### 用户价值

让用户可以将一次 PR 作为独立任务进行分析和管理。

### 输入内容

| 字段     | 是否必填 | 说明                   |
| ------ | ---- | -------------------- |
| PR 标题  | 必填   | 简要描述本次 PR            |
| PR 描述  | 选填   | 开发者对本次改动的说明          |
| 代码变更内容 | 必填   | 本次 PR 的代码 diff 或修改片段 |
| 关联项目   | 选填   | 所属项目名称               |
| 目标分支   | 选填   | 本次 PR 要合并到的分支        |
| 开发者名称  | 选填   | PR 提交者               |

### 页面元素

| 元素           | 说明              |
| ------------ | --------------- |
| PR 标题输入框     | 输入本次 PR 标题      |
| PR 描述输入框     | 输入本次 PR 背景和说明   |
| 代码变更输入区      | 输入代码 diff 或代码片段 |
| 开始 Review 按钮 | 提交任务并开始分析       |
| 取消按钮         | 放弃创建任务          |

### 交互规则

1. PR 标题不能为空。
2. 代码变更内容不能为空。
3. 如果必填项为空，点击“开始 Review”时提示用户补充。
4. 用户提交后，系统进入 Review 生成状态。
5. Review 完成后，自动进入结果页。

---

## 7.2 PR 摘要生成

### 功能说明

系统根据用户输入的 PR 信息，自动生成本次 PR 的摘要。

### 用户价值

帮助 Reviewer 快速了解本次 PR 的主要内容，减少阅读成本。

### 输出内容

PR 摘要应包含：

1. 本次 PR 的主要目的。
2. 修改了哪些功能或模块。
3. 涉及的关键文件或代码区域。
4. 可能影响的业务流程。
5. 是否涉及测试、配置、接口、权限、安全等内容。

### 示例

```text
本次 PR 主要修改了用户登录模块，新增了 token 刷新逻辑，并调整了登录状态校验流程。
主要影响 AuthService 和 UserController。
本次改动涉及认证逻辑，建议重点关注 token 过期处理和异常场景。
```

### 验收标准

1. 系统能够根据输入内容生成简洁摘要。
2. 摘要应避免只复述代码。
3. 摘要应突出改动目的和影响范围。
4. 摘要内容应使用自然语言表达，便于非提交者理解。

---

## 7.3 风险等级评估

### 功能说明

系统根据 PR 的改动内容，对本次 PR 给出风险等级。

### 风险等级

| 风险等级 | 含义                         |
| ---- | -------------------------- |
| 低风险  | 改动较小，主要为样式、注释、文案或轻量逻辑修改    |
| 中风险  | 涉及业务逻辑、接口、数据处理或多个文件修改      |
| 高风险  | 涉及权限、安全、支付、删除、并发、核心流程等关键逻辑 |

### 风险评估维度

| 维度     | 说明                 |
| ------ | ------------------ |
| 改动规模   | 修改文件数量、代码变更量       |
| 模块重要性  | 是否涉及核心业务模块         |
| 代码复杂度  | 是否包含复杂逻辑、条件分支、异常流程 |
| 安全敏感性  | 是否涉及权限、认证、数据安全     |
| 测试完整性  | 是否补充或修改测试          |
| 规则命中情况 | 是否违反团队配置规则         |

### 输出形式

```text
风险等级：高

原因：
1. 本次 PR 修改了用户认证逻辑。
2. 代码中新增了 token 校验流程。
3. 未发现对应测试用例。
4. 异常场景处理不完整。
```

### 验收标准

1. 每个 Review 报告必须包含风险等级。
2. 风险等级必须附带原因说明。
3. 风险等级不能只给结论，必须解释依据。
4. 风险等级应与问题列表保持一致。

## 7.4 代码问题识别

### 功能说明

系统根据 PR 变更内容，识别代码中可能存在的问题。

### 问题类型

MVP 阶段支持以下问题类型：

| 类型     | 说明                  |
| ------ | ------------------- |
| 逻辑问题   | 条件判断错误、流程不完整、边界情况遗漏 |
| 异常处理问题 | 缺少错误处理、异常被忽略        |
| 安全问题   | 权限校验不足、敏感信息暴露、输入未校验 |
| 性能问题   | 不必要的循环、重复计算、低效查询    |
| 可维护性问题 | 代码重复、命名不清晰、函数过长     |
| 测试缺失   | 修改核心逻辑但未补充测试        |
| 规范问题   | 不符合团队配置的 Review 规则  |

### 每个问题包含的信息

| 字段     | 说明             |
| ------ | -------------- |
| 问题标题   | 简短描述问题         |
| 问题类型   | 逻辑、安全、性能等      |
| 严重程度   | 高 / 中 / 低      |
| 问题说明   | 解释为什么这是问题      |
| 相关代码位置 | 指出问题对应的代码片段或文件 |
| 修改建议   | 给出可执行的优化方向     |
| 置信度    | AI 对该问题判断的可信程度 |

### 严重程度定义

| 严重程度 | 说明                  |
| ---- | ------------------- |
| 高    | 可能导致线上故障、安全漏洞、数据错误  |
| 中    | 可能影响稳定性、可维护性或部分异常场景 |
| 低    | 主要影响代码风格、可读性或小范围优化  |

### 示例

```text
问题标题：缺少空值判断

问题类型：异常处理问题
严重程度：中
相关位置：UserService.getUserAvatar()

问题说明：
当前代码直接访问 user.profile.avatarUrl。
如果 profile 为空，可能导致运行时异常。

修改建议：
在访问 profile 前增加空值判断，或在用户初始化时保证 profile 一定存在。
```

### 验收标准

1. 系统能够输出问题列表。
2. 每个问题必须包含问题说明和修改建议。
3. 问题应按严重程度排序。
4. 高风险问题应优先展示。
5. 问题描述应尽量具体，避免空泛建议。

---

## 7.5 修改建议生成

### 功能说明

系统针对识别出的问题，给出修改建议。

### 用户价值

帮助开发者理解问题并快速改进代码，而不是只指出错误。

### 修改建议要求

1. 建议应具体。
2. 建议应说明修改方向。
3. 建议应避免过度泛化。
4. 建议应尽量结合当前代码上下文。
5. 建议应明确说明为什么要这样改。

### 建议类型

| 类型     | 示例            |
| ------ | ------------- |
| 补充判断   | 增加空值检查、边界条件判断 |
| 补充异常处理 | 捕获异常、返回错误信息   |
| 优化结构   | 拆分函数、减少重复代码   |
| 补充测试   | 增加单元测试或异常场景测试 |
| 加强安全   | 增加权限校验、输入校验   |
| 改善可读性  | 优化命名、补充必要注释   |

### 示例

```text
建议：
当前函数同时处理参数校验、数据库查询和结果转换，职责较多。
建议将参数校验逻辑拆分到独立函数中，使主流程更清晰，也方便后续单元测试。
```

### 验收标准

1. 每个中高风险问题必须包含修改建议。
2. 建议应与问题直接相关。
3. 建议不能只写“建议优化”这类空泛内容。
4. 建议应让开发者知道下一步怎么做。

## 7.6 Review 报告详情页

### 功能说明

系统将一次 AI Review 的结果整理为结构化报告，供开发者和 Reviewer 查看。

### 页面内容

Review 报告详情页应包含：

1. PR 基本信息
2. Review 状态
3. PR 摘要
4. 风险等级
5. 问题统计
6. 问题列表
7. 修改建议
8. 用户反馈入口

### 页面结构

```text
PR Review 报告
├── PR 基本信息
├── AI 摘要
├── 风险等级
├── 问题统计
├── 高风险问题
├── 中风险问题
├── 低风险问题
└── 用户反馈
```

### 问题统计

| 指标     | 说明                |
| ------ | ----------------- |
| 总问题数   | 本次 Review 发现的问题总数 |
| 高风险问题数 | 严重程度为高的问题数量       |
| 中风险问题数 | 严重程度为中的问题数量       |
| 低风险问题数 | 严重程度为低的问题数量       |
| 规则命中数  | 命中团队规则的问题数量       |

### 验收标准

1. 用户可以清晰看到本次 PR 的整体结论。
2. 用户可以快速定位高风险问题。
3. 用户可以查看每个问题的详细说明。
4. 用户可以看到每个问题对应的修改建议。
5. 报告结构应清晰，适合比赛展示。

## 7.7 基础 Review 规则配置

### 功能说明

项目负责人可以配置基础 Review 规则，系统在 Review 时根据规则进行检查和提示。

### 用户价值

让 AI Review 更符合团队自己的代码规范，而不是只给通用建议。

### MVP 支持的规则类型

| 规则类型   | 示例                             |
| ------ | ------------------------------ |
| 必须补充测试 | 修改核心模块时必须补充测试                  |
| 禁止提交内容 | 禁止出现 console.log、debugger、TODO |
| 文档同步要求 | 修改接口时必须更新接口文档                  |
| 安全规则   | 涉及权限逻辑时必须说明安全影响                |
| 命名规范   | 函数名、变量名应符合团队规范                 |
| 模块约束   | Controller 层不能直接操作数据库          |

### 规则字段

| 字段   | 是否必填 | 说明           |
| ---- | ---- | ------------ |
| 规则名称 | 必填   | 规则的名称        |
| 规则描述 | 必填   | 规则的具体说明      |
| 规则类型 | 必填   | 测试、规范、安全、文档等 |
| 严重程度 | 必填   | 高 / 中 / 低    |
| 是否启用 | 必填   | 启用或关闭该规则     |

### 规则示例

```text
规则名称：修改核心逻辑必须补充测试
规则类型：测试要求
严重程度：中
规则描述：如果 PR 修改了登录、支付、权限等核心逻辑，需要补充对应测试用例。
状态：启用
```

### 验收标准

1. 用户可以新增规则。
2. 用户可以编辑规则。
3. 用户可以启用或停用规则。
4. Review 报告中应显示命中的规则。
5. 规则命中的问题应出现在问题列表中。

## 7.8 历史 Review 记录

### 功能说明

用户可以查看历史创建过的 PR Review 任务。

### 用户价值

方便用户回顾过去的审查结果，也便于项目负责人观察团队代码质量变化。

### 列表字段

| 字段        | 说明                   |
| --------- | -------------------- |
| PR 标题     | Review 任务标题          |
| 所属项目      | 关联项目                 |
| 创建人       | 创建任务的用户              |
| 创建时间      | 任务创建时间               |
| 风险等级      | 本次 Review 的风险等级      |
| 问题数量      | 本次发现的问题总数            |
| Review 状态 | 待分析 / 分析中 / 已完成 / 失败 |

### 支持操作

| 操作        | 说明              |
| --------- | --------------- |
| 查看详情      | 进入 Review 报告详情页 |
| 删除记录      | 删除历史 Review 记录  |
| 重新 Review | 基于当前内容重新生成报告    |

### 验收标准

1. 用户可以看到历史 Review 任务列表。
2. 用户可以进入某个任务的详情页。
3. 用户可以根据风险等级快速判断任务情况。
4. 用户可以区分任务状态。

## 7.9 用户反馈

### 功能说明

用户可以对 AI 输出的问题和建议进行反馈。

### 用户价值

帮助判断 AI Review 的有效性，也为后续产品优化提供依据。

### 反馈类型

| 反馈   | 含义          |
| ---- | ----------- |
| 有用   | 该建议对用户有帮助   |
| 无用   | 该建议没有实际价值   |
| 误报   | AI 判断的问题不存在 |
| 已采纳  | 用户已根据建议修改   |
| 暂不处理 | 用户认为当前无需修改  |

### 反馈位置

用户可以对每条问题建议单独反馈。

### 验收标准

1. 每条问题建议旁边应有反馈入口。
2. 用户可以标记建议是否有用。
3. 用户可以标记问题是否为误报。
4. Review 报告中可以显示反馈状态。

---

## 7.12 PR 链接抓取

### 功能说明

用户输入 GitHub PR URL 后，系统自动抓取 PR 标题、描述、diff 内容，并填充到新建 Review 表单中。

### 输入内容

| 字段 | 是否必填 | 说明 |
| ---- | ---- | ---- |
| GitHub PR URL | 必填 | 格式为 `https://github.com/owner/repo/pull/123` |

### 返回内容

| 字段 | 说明 |
| ---- | ---- |
| title | PR 标题 |
| description | PR 描述（可能为空） |
| diff_content | PR 代码 diff（最长 50000 字符） |
| project_name | 仓库全名（如 `octocat/hello-world`） |
| target_branch | PR 目标分支（base ref） |
| developer_name | PR 提交者 GitHub 用户名 |

### 验收标准

1. 输入有效 GitHub PR URL 后，返回标题、描述、diff 及项目/分支/开发者信息。
2. URL 解析支持尾部路径（如 `/files`）和锚点片段。
3. GitHub API 不可达时返回明确错误，含 404、403、429 等状态。
4. diff 超过 50000 字符时自动截断。

## 11. 非功能需求

## 11.1 易用性

1. 用户应能在较短时间内完成一次 Review 任务创建。
2. Review 报告结构应清晰，避免大段无重点文本。
3. 高风险问题应突出展示。
4. 修改建议应便于开发者理解。
5. 页面用词应偏工程化，避免过度营销化表达。

---

## 11.2 准确性

1. AI 输出的问题应尽量基于用户提供的代码内容。
2. 对不确定的问题应降低严重程度或标明置信度。
3. 不应将猜测性内容作为确定结论。
4. 高风险问题应有明确理由。
5. 修改建议应与问题内容一致。

---

## 11.3 可解释性

1. 风险等级必须说明原因。
2. 每个问题必须说明为什么存在风险。
3. 修改建议应说明修改目的。
4. 对团队规则命中的问题，应展示命中的规则名称。

---

## 11.4 安全性

1. 用户提交的代码内容应作为敏感信息处理。
2. Review 报告不应公开给无关人员。
3. 系统不应在未经用户确认的情况下自动修改代码。
4. 系统不应自动合并 PR。
5. 系统不应输出与代码无关的敏感推断。

## 13. 验收标准

MVP 版本整体完成后，应满足以下验收条件：

1. 用户可以创建 PR Review 任务。
2. 用户可以输入 PR 标题、描述和代码变更内容。
3. 系统可以生成 PR 摘要。
4. 系统可以给出风险等级。
5. 系统可以输出问题列表。
6. 每个问题包含类型、严重程度、说明和修改建议。
7. 用户可以查看完整 Review 报告。
8. 用户可以配置基础 Review 规则。
9. Review 报告可以展示规则命中结果。
10. 用户可以查看历史 Review 记录。
11. 用户可以对 AI 建议进行反馈。
12. 系统不会自动修改代码。
13. 系统不会自动合并 PR。
14. Review 结果应清晰、结构化、可展示。

## 15. 典型使用场景

## 场景一：开发者提交 PR 前自检

### 用户

开发者

### 场景描述

开发者完成一个登录功能修改，在提交给 Reviewer 前，先使用 AI PR Review 助手进行自检。

### 用户操作

1. 创建 Review 任务。
2. 输入 PR 标题：优化用户登录逻辑。
3. 输入 PR 描述：新增 token 刷新机制。
4. 输入代码变更内容。
5. 点击开始 Review。
6. 查看 AI 报告。
7. 根据建议补充异常处理和测试用例。

### 用户收益

开发者可以在正式提交人工 Review 前提前发现问题，减少返工。

---

## 场景二：Reviewer 快速理解 PR

### 用户

Reviewer

### 场景描述

Reviewer 收到一个改动较大的 PR，不确定应该优先看哪些内容。

### 用户操作

1. 打开 AI Review 报告。
2. 阅读 PR 摘要。
3. 查看风险等级。
4. 优先查看高风险问题。
5. 结合 AI 建议进行人工判断。

### 用户收益

Reviewer 不需要从头阅读所有代码，可以快速定位重点区域。

---

## 场景三：项目负责人统一团队规范

### 用户

项目负责人

### 场景描述

团队希望所有成员修改核心模块时必须补充测试。

### 用户操作

1. 进入规则配置页。
2. 新增规则：修改核心模块必须补充测试。
3. 设置规则类型：测试要求。
4. 设置严重程度：中。
5. 启用规则。
6. 后续 Review 报告中展示规则命中情况。

### 用户收益

团队可以将 Review 规范沉淀到产品中，降低人工提醒成本。

## 16. 需求总结

AI PR Review 助手 MVP 版本的核心是：

> 让用户输入一次 PR 代码变更，系统自动生成结构化代码审查报告，包括 PR 摘要、风险等级、问题列表和修改建议。

MVP 阶段应优先证明三个价值：

1. **帮助开发者提前发现问题。**
2. **帮助 Reviewer 更快理解 PR。**
3. **帮助团队形成统一 Review 标准。**

因此，MVP 不追求功能复杂，而应聚焦在完整闭环：

```text
创建 Review 任务
    → 输入 PR 变更内容
    → AI 生成摘要
    → AI 识别问题
    → AI 给出建议
    → 用户查看报告
    → 用户反馈结果
```

这个版本已经具备清晰的产品价值、完整的用户路径和较好的比赛展示效果。
